## New findings

### R2-1 (major): The published "immutable approval ref" is unreachable from any public ref

`docs/releases/v0.1.md` on public `main` records:

> Approval ref: `e9fdfc06f2c599b00210eba17f66985bca4c002e`

Verified: that commit is reachable **only** from the local branch
`archive/agent-work-v0.1-pre-squash`. No remote branch contains it
(`git branch -r --contains e9fdfc0` is empty), and the pre-squash history was
never pushed. A member of the public cloning the repository cannot resolve the
approval ref at all — the immutable anchor of the release's provenance chain
dangles for exactly the audience it was published to convince.

Compounding it, `audit_release_v0_1.py` verifies only that the *document*
contains a 40-hex string in the right lines; it never runs
`git cat-file -e <ref>`, so the audit passes in any clone regardless of
whether the ref exists anywhere.

Fix options, in order of preference: push a tag (e.g. `v0.1-approval`)
pointing at `e9fdfc0`, or push the archive branch; and make the v0.1 audit
resolve the ref when the object store contains it, reporting "unresolvable
here" otherwise instead of silently not checking.

### R2-2 (major): The incomplete-proof gate cannot catch the normal form of `sorry`

`validate_current_state.py` scans with:

```python
forbidden = re.compile(r"^\s*(sorry|admit|axiom)\b", re.MULTILINE)
```

Verified by direct test: `theorem x : P := sorry`, `exact sorry`, and
`have h : Q := by sorry` — the forms `sorry` actually takes in Lean code —
all pass this regex; only a `sorry` alone at the start of a line is caught.
Meanwhile `lake build` treats `sorry` as a *warning* ("declaration uses
'sorry'"), exits 0, and `lean-action` passes. Failure scenario: a PR adds
`:= by sorry` to any released file; every CI gate is green; the README claim
"Released Lean files contain no `sorry`" becomes false on protected `main`.

Fix: match `\bsorry\b`/`\badmit\b` anywhere (excluding comments/docstrings if
needed), or fail CI on the build warning (grep the build log, or set the lib's
`leanOptions` to promote the warning), preferably both. Note the current tree
is clean — verified by unanchored grep — the finding is about the gate, not
the code.

### R2-3 (moderate): The worked halting example is outside every build that CI runs

`AISafetyAtlas.lean` no longer imports
`Survey/BrcicYampolskiy/HaltingExample.lean`, the Examples modules don't
either, and Lake builds only the root import closure plus the two example
targets CI names. Verified empirically: with the full cache warm, an explicit
build of the module compiled it *fresh* — nothing else builds it. Yet the
immutable v0.1 evidence row says "Compiling fixed-input halting example …
Passed", and `validate_current_state.py` checks only that the file *exists*.
Failure scenario: a Mathlib bump breaks the example; CI stays green; the
repository ships a non-compiling file that its release evidence calls
compiling. Fix: add the module to the CI build step (one word) or import it
from `Examples/PublicAPI.lean`.

### R2-4 (moderate): open-work.md still tells contributors No Free Lunch is unformalized territory — the repository's own evidence now says otherwise

`docs/guide/open-work.md`: "Treat No Free Lunch as a high-novelty candidate after a
fresh primary-source and formal-library search. The reports' claim that no
formalization exists is useful search direction…"

But the regenerated search (this same batch of commits) surfaced AFP
`No_Free_Lunch_ML` for both BY-020 and BY-021, and the registry notes record
it as a candidate awaiting statement-level verification. The contributor-facing
document and the evidence now disagree: a contributor following open-work.md
would begin "high-novelty" NFL work whose canonical prior sits one directory
over — precisely the duplication the parsimony policy exists to prevent.
Fix: rewrite the open-work NFL item as "triage AFP `No_Free_Lunch_ML` at
statement level against BY-020/BY-021", which is also the obvious next
coverage win.

### R2-5 (minor): Operational claims with no in-repo evidence, including a "no findings" review claim

STATE.md and ROADMAP assert: "Protected `main` rules configured", "Repository
remained private until approval was recorded", and "independent code re-review
reports no findings". None of these have evidence artifacts in the repository
(branch-protection and visibility history live in GitHub settings; the
re-review has no report file). The "no findings" claim in particular sits
awkwardly next to this review's findings list and next to the F-01–F-05 audit
the same day. Recommendation: link each operational claim to where its
evidence can be checked (settings screenshot/API output under `reviews/` or
`docs/`), and name the reviews that were performed with pointers to their
reports.

### R2-6 (observation): `Verification.rice` is sound and honestly framed; whether it clears the project's own "not an AI-themed alias" bar is a genuine open question — the pending review is not ceremonial

The proof was hand-checked: the codes-set is extensional by construction,
`Nontrivial` supplies the nonempty/nonuniversal witnesses, and the conclusion
follows from `rice_code_iff` correctly. The registry classifies it `BRIDGE`
under BY-012 with `ai_bridge_status: HUMAN_REVIEW`; README explicitly
disclaims application to particular AI systems. All of that is the methodology
working as designed.

The adversarial observation: mathematically the theorem is a short corollary
of `rice₂` — its entire content is the vocabulary (`BehavioralProperty`,
`Holds`, `HasVerifier`) plus two witnesses. ROADMAP's own success criterion is
"a reusable bridge theorem, not an AI-themed alias for Rice's theorem". The
difference between this and an alias is the explicit model statement — which
is real but thin until a downstream result *consumes* the interface. STATE's
next-task 2 ("specify one concrete downstream AI-safety use before extending
the verification API; otherwise revise or keep the bridge local") is exactly
the right test; the maintainer review should treat it as load-bearing.

## What held up under attack

- **The iteration loop works.** Every round-1 finding was fixed, and fixed
  properly — verified against the artifacts, not the commit messages. The
  search regeneration in particular turned a criticized evidence set into a
  discovery (`No_Free_Lunch_ML`, AFP Löb files, HOL-Light GL).
- **Provenance pins remain solid.** Manifest Mathlib rev, registry pin, and
  toolchain tag all agree; the vNM upstream exists at its pinned revision; the
  vendored Arrow delta is now fully disclosed and matches the actual diff.
- **The generated-views architecture removes a whole failure class.** Status
  tables, README scope, and declaration checks are derived from the registry
  and drift-checked in CI (`--check` verified passing).
- **The new public-API contract is real.** `Examples/PublicAPI.lean` fixes
  statement shapes; `Examples/Registry.lean` covers all 8 registry
  declarations; both compile through the single root import.
- **Honesty discipline held under publication pressure.** The published README
  states 3/44 coverage and 0 reviewed bridges in a generated block; the vNM
  reproduction records its packaging caveat and linter warnings instead of
  hiding them; "Reviewed AI-system bridge theorems: 0" appears in the release
  evidence itself.
